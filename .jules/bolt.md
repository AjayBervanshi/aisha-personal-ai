## 2024-05-18 - [Batch Updates to Avoid N+1 Problem with Supabase]
**Learning:** O(N^2) memory analysis loops (like cosine similarity computations) with individual synchronous database updates for each match result in severe network IO overhead and N+1 query patterns. Supabase has PostgREST URL length limits, meaning batched `.in_()` statements should chunk requests.
**Action:** Always accumulate entity IDs inside loops and apply updates in batches outside the loop using `.in_()` (chunked at max 100 items) to prevent blocking the main thread and ensure scalability over time.## 2024-05-18 - [Limit Supabase Network Payloads for Counting]
**Learning:** By default, PostgREST queries that fetch row counts using `count="exact"` will download all matching rows (O(N) data overhead) unless limited. PostgREST handles the `count` calculation completely independent of the returned data limit.
**Action:** Always append `.limit(1)` to Supabase `.select('id', count='exact')` queries when you only need the count property and not the actual row data.
